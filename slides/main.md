# Max 20x — 왜 Pro 대신 Max 인가

<!-- timing: 90 -->

**2개월 실사용 기준 개인 관찰 — 모델 버전에 따라 언제든 바뀔 수 있음**

- **아키텍팅 능력 탁월** — 다른 AI 도구 대비 전체 구조를 먼저 그림. 부분 패치보다 설계 수준의 접근을 먼저 제안
- **주변 코드와 유사한 패턴 유지** — 이상한 스타일·패턴이 섞이지 않음. 기존 코드베이스 관용어를 따름
- **수정량이 적음** — 출력이 기존 패턴에 가까워서 리뷰·수정 부담이 줄었음
- **리뷰 편의성** — 제 경험상 타사 AI 도구 대비 저한테는 리뷰가 수월했음 (개인적 의견)

# 한계 1: 검증은 여전히 개발자 몫

**개인 원칙**

- **"AI 출력은 항상 초안"** — 그대로 쓰는 코드는 없음

- **"빌드·테스트·리뷰는 여전히 개발자의 일"** — AI 가 대신할 수 없는 책임

# 한계 4: 생성된 코드가 느릴 수 있음

**AI 가 만든 코드 ≠ 성능이 나오는 코드**

- build → test → build → test 피드백 루프에 집중하다 보면 **AI agent 는 성능을 신경 쓰지 않음**
- 기능만 맞는 코드, 비효율적 자료구조·알고리즘 선택 사례 관찰됨
- 성능 측정 벤치마크를 피드백 루프에 추가하면 성능까지 고려 가능 — 아직 발표자는 경험해본 적 없음
- **목표 성능치를 애초에 주는 것부터 도전적** — "얼마나 빨라야 하는가" 가 사람에게도 불분명한 경우 많음

<div class="caveat">개인 사용 경험의 한계일 수 있음. 기능 구현 위주 루프만 돌려봤기 때문. 성능 크리티컬 영역은 여전히 사람이 직접 튜닝 중.</div>

# Spec Driven Development (SDD)

**요구사항·스펙을 먼저 글로 명확히 작성한 뒤 구현에 들어가는 개발 방식**

- AI와 협업할 때 요구사항을 먼저 명확히 적어두면 결과 품질이 올라가는 걸 체감했음
- "무엇을 만들어야 하는가"를 글로 정리하는 것이 AI 지시에 더 효과적
- "요구사항을 먼저 글로 쓰는 것"이 AI 활용 ROI를 높이는 데 도움이 됐다는 경험 공유

<!-- timing: 45 -->

---

# SDD 의 핵심 — SSOT 와 JIRA 이슈

**SSOT (Single Source of Truth) 를 만드는 것이 중요**

- 스펙이 여러 곳에 흩어져 있으면 AI 도 사람도 혼란 — 단일 원본이 있어야 함
- **JIRA 이슈를 최대한 자세히 작성**하는 것이 중요 — JIRA 이슈 자체가 SSOT 역할
- 배경, 목적, 요구사항, 예상 동작, 예외 케이스까지 글로 남길수록 재사용 가치가 올라감

**앞으로의 방향성**

- Claude Code 가 **JIRA 이슈를 직접 읽고 구현을 자동 수행**하는 흐름으로 갈 것으로 예상
- 이슈 작성에 들이는 시간이 **곧 구현 품질** 로 환산되는 구조
- "잘 쓴 이슈 = 잘 구현된 코드" 의 연결이 더 강해짐

<!-- timing: 75 -->

# Test Driven Development (TDD)

**실패하는 테스트를 먼저 작성하고, 그 테스트를 통과시키도록 구현하는 개발 방식**

- 테스트를 먼저 쓰면 AI가 그 테스트를 통과하는 구현을 맞춰 생성하는 방식이 잘 동작했음
- 검증 기준이 먼저 있으면 AI 출력의 품질 판단이 쉬워졌음

# 꿀팁 — "프롬프트를 직접 짜지 말고, 추천받아라"

**의도만 던지고 프롬프트는 AI 가 구성하게 하기**

> *"X 를 하고 싶다. 지금 사용 가능한 skill 과 명령어를 조합해서 가장 좋은 프롬프트를 추천해줘."*

**왜 직접 작성보다 나은가**

- Claude Code 는 **현재 설치된 skill / slash command / MCP 도구**를 모두 알고 있음
- 자기 자신이 가진 도구를 조합해 **상황 맞춤형 프롬프트 + 실행 가능 옵션**을 동시에 제시
- 유저는 후보 중에서 고르기만 하면 됨 → **프롬프트 엔지니어링 부담 제거**

**왜 능숙한 사용자에게도 도움이 되나**

- 직접 `/deep-interview`, `/ralph`, `/autopilot` 을 일일이 실험할 필요 없음
- 모르고 지나칠 뻔한 **자기 환경의 skill 조합**을 발견하게 됨
- "프롬프트 엔지니어링" 의 2-step 진화형 — **의도 → 추천 프롬프트 → 실행**

<span class="qualifier">초심자·고수 모두에게 추천 — 그냥 "추천 프롬프트를 작성해줘" 한 줄이면 됨</span>

# 자동화 자동화 — 시행착오 자체를 자산으로 굳히기

**핵심 패턴**

1. 우선 **하고 싶은 일을 Claude 에 명령** — 처음에는 부정확해도 OK
2. AI 가 **시행착오를 거쳐 성공**할 때까지 사람이 옆에서 코칭
3. 성공한 직후 **"방금 한 작업을 다음에도 반복할 수 있게 skill 로 만들어줘"** 라고 지시
4. 다음번에는 시행착오 없이 **skill 호출 한 번으로 같은 결과**

**왜 이게 게임체인저인가**

- 첫 시도의 비용을 **미래의 모든 반복에 분할 상환**
- 사람이 "어떻게 하라" 가 아닌 **"무엇을 원한다"** 만 말해도 AI 가 절차를 만들어 저장
- 성공한 시행착오를 **잊어버리지 않게** AI 자신에게 노트로 남기게 하는 셈

**대표 예시**

- `/learner` — 현재 세션의 작업을 분석해 **반복 가능한 skill 초안**으로 추출
- `/skill-creator` — 그 초안을 정식 skill 로 등록 → 곧바로 재사용 가능

<span class="qualifier">"자동화하는 작업을 자동화" 하는 메타 워크플로 — 한 번 익히면 평생 재산</span>

# 끝으로 — 사람에게 친화적인 업무가 AI 에게도 친화적

**신규 입사자와 AI 는 같은 입장에서 프로젝트를 바라본다**

- 신규 입사자가 빠르게 업무를 파악할 수 있도록 정리할수록 → **AI 의 업무 효율도 같이 올라감**
- 반대로 AI 효율을 높이려고 정리한 문서·스크립트는 → **신규 입사자 온보딩을 그대로 가속**
- **Win-win** — 한쪽을 위한 투자가 양쪽 모두를 돕는 구조

**구체적 예시**

- `ctp.sh --help` 메시지, `README`, 빌드 스크립트 주석 — AI 가 이걸 먼저 읽고 업무를 파악함
- 이 문서들을 개선하면 **신규 입사자의 업무 파악 속도**도 자연스럽게 빨라짐
- 결국 "AI 친화적 개선" 과 "사람 친화적 개선" 은 **같은 방향**

# 암묵지를 명시화하면 AI 도 사람도 빨라진다

**에러 발생 시 "어디를 봐야 하는지" 가 문서로 드러나 있어야 함**

- 에러가 나면 **즉시 원인을 파악할 수 있거나**, **어디서 에러 메시지를 찾아볼 수 있는지** 피드백 주는 것이 중요
- 예: "서버가 갑자기 죽으면 `$CUBRID/log/server/<db_name>_stdout.log` 를 먼저 본다" 같은 지식
- 이런 내용이 **팀원 머릿속에만 있는 암묵지**로 남아 있으면 AI 는 접근할 수 없음

**암묵지 → 명시지 전환**

- README, 트러블슈팅 가이드, `--help`, 로그 위치 주석 등으로 **글로 적어두기**
- 암묵지가 많을수록 AI 효율은 떨어지고, 신규 입사자 진입 장벽은 올라감
- 반대로 명시화할수록 **AI·신입·기존 개발자 모두가 같이 빨라짐**

# AI 자동화의 가장 큰 적 — Hang · 무한 루프 · 끝을 모르는 작업

**AI 가 무작정 대기하는 상황이 생산성을 가장 크게 갉아먹음**

- 타임아웃이 길수록 **feedback loop 전체가 느려짐** — 한 번 막히면 세션 전체가 지연
- "끝날 때까지 기다린다" 는 것이 가장 비싼 비용

**대응 전략**

- **iteration 횟수나 경과 시간에 warning** 을 걸어두기 — N 회 초과·M 분 경과 시 알림
- AI 에게 **"10 분이 지나면 다른 방법을 찾아보라"** 고 명시적으로 지시
- 외부 명령 실행 시에는 **timeout 을 짧게** 걸고, 실패 시 대안 경로를 시도하도록 설계

# 환경 전제 검증 — Skill 앞단의 prerequisite 패턴

**현재 발표자 운영 방식**

- 각 Skill 의 **맨 앞에 `prerequisite` / `requirements` 블록**을 둠

**왜 이렇게 하나**

**더 나은 방식 모색 중**

- 단발성 검증 → **세션 시작 hook**, MCP 도구, 혹은 자동 복구(branch checkout / rebuild)까지
- 팀·본부 차원의 **표준 prerequisite 라이브러리** 가 있으면 더 효과적

# 연구개발본부 업무 자동화 후보 — Git / GitHub 영역

**① Git 일상 흐름 자동화**

- `git stage → add → commit → push` 한 번에 — 변경 분석으로 **커밋 메시지 자동 작성**
- 팀 컨벤션(prefix, JIRA 키, 영문/한글 요약)을 학습시켜 **일관된 메시지 품질**

**② Git Merge Conflict 해결 — 컨텍스트 기반**

- "develop 쪽 의도가 우선" / "특정 모듈은 feature 우선" 같은 **자연어 지시를 받아** AI 가 hunk 단위로 판단
- 단순 `--ours` / `--theirs` 보다 **세분화된 conflict 해결 정책** 적용 가능
- 충돌 패턴이 반복되면 skill 로 굳혀 매번 사람이 일일이 풀 필요 없어짐

**③ gh CLI / API 자동화**

- PR 생성·draft 전환, **리뷰어 자동 등록·re-request**, 댓글·대댓글, 머지 후 정리
- **Greptile 봇 응답 일괄 처리** — 이미 발표자 운영 중 (skill 화)

# 연구개발본부 업무 자동화 후보 — JIRA / 문서 영역

- 한 번 skill 화하면 **본부 전체가 공유** 가능
