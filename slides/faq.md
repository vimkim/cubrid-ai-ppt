# FAQ B2: 프롬프트 좀 보여주세요

**특별한 프롬프트 엔지니어링보다는, 맥락을 충분히 주고 작은 질문을 여러 번 던지는 방식이 잘 먹혔습니다.**

- `"이 파일/함수가 뭘 하나요"` — 탐색 시작점
- `"이 에러 로그 원인이 어디일까요, 후보 3개만"` — 원인 범위 좁히기
- `"이 변경 기반으로 PR description 초안"` — 문서화 시작점
- 긴 프롬프트보다 짧고 구체적인 질문들이 더 일관된 답을 줬습니다

# FAQ B3: 한글/UTF-8 깨지는 건 어떻게?

**대체로 문제없이 됩니다. 몇 가지 포인트만 조심하면 됩니다.**

- 터미널 로케일이 `ko_KR.UTF-8` 또는 `en_US.UTF-8`이면 OK
- 코드 블록 안 한글 주석/문자열도 정상 처리
- 다만 출력 폭 계산에서 한글이 1칸으로 카운트되는 UI 이슈가 간헐적으로 있음 (Claude 측 이슈, 내용은 정확)
- EUC-KR 인코딩 파일은 주의 — Claude는 UTF-8을 가정함

# FAQ B4: 비용/라이센스는?

**Max 플랜은 개인 월정액. 팀 단위 도입은 별도 검토 필요.**

- Max 플랜 = Claude.ai 개인 계정 월정액 ($100 또는 $200)
- Claude Code는 API와 다르게 Max 플랜 안에 포함
- 팀 도입 시: 사내 정책, 계정 관리, 비용 분담 검토 필요
- Anthropic 별도 팀/엔터프라이즈 옵션 있음 (별도 문의)

# FAQ B5: 사내 보안/IP 유출 리스크는?

**코드가 외부 모델로 전송된다는 점은 분명합니다. 사내 정책 확인이 전제입니다.**

- Claude Code는 프롬프트+파일 내용을 Anthropic 서버로 전송
- Anthropic의 데이터 처리 정책은 공개되어 있으나, 사내 기준과 맞는지 확인 필요
- 민감 코드/비밀키가 포함된 파일은 의도적으로 제외해야 함
- 전사 도입 시 보안팀 검토 권장

# FAQ B6: Copilot/Cursor와 비교하면?

**도구마다 강점이 다릅니다. 본 발표는 Claude Code 2개월 개인적 경험 공유이므로 공격적 비교는 피하겠습니다.**

- Copilot: IDE 통합, 코드 자동완성 강점
- Cursor: 에디터 자체가 AI 통합, 편집 흐름 강점
- Claude Code: 터미널, 파일 시스템 접근, 긴 문맥/탐색 강점
- "어느 것이 우리 팀에 맞는가"는 실험 + 비교가 최선

# FAQ B7: 환각이나 틀린 코드 얼마나 자주?

**낯선 코드일수록 자주 납니다. 검증 습관이 필수입니다.**

- 익숙한 언어/라이브러리: 드문 편
- 낯선 서브시스템/내부 API: 잦은 편 (함수 이름/필드 추측)
- 항상 `빌드 + 실행 + 리뷰`로 확인
- 개인적으로 의미 있는 작업 중 일부는 큰 수정이 필요했음 (주관적)

# FAQ B8: 학습 곡선은?

**도구 자체는 빨리 배웁니다. '잘 쓰는' 법은 꾸준한 실험이 답입니다.**

- 설치/기본 명령: 30분 안에 시작 가능
- 프롬프트 감각: 1~2주 실사용하며 자연스럽게
- 한계 인식: 환각을 몇 번 겪어봐야 진짜로 체감
- 팀 안에서 팁 공유하면 학습 속도가 빨라짐

# FAQ B9: Pro로도 충분한가?

**개인 워크플로우에 따라 다릅니다.**

- Pro 플랜도 일반적인 사용에는 충분할 수 있음
- 대규모 코드베이스를 긴 대화로 반복 탐색하다 보니 Pro 사용량 한도가 금방 소진됐음
- Max 20x로 넘어간 주된 이유는 **컨텍스트 길이와 사용 한도** 때문
- heavy 사용자라면 Max 20x 가성비가 훨씬 좋음 — API 대비 10~36배 절감 가능

# FAQ B11: "생성 코드가 느릴 수 있다"는 구체적으로?

**AI 가 동작은 맞춰도 성능은 신경 쓰지 않음**

- build → test 피드백 루프에만 집중하면 알고리즘·자료구조 선택이 비효율적인 경우가 있었음
- 정답 기능은 나오지만 불필요한 반복 순회, 중복 락 획득 등이 관찰됨
- 성능 벤치마크를 피드백 루프에 포함하면 개선 여지 있음 — 아직 직접 시도해본 적 없음
- 목표 성능치를 AI 에게 숫자로 주는 것부터가 어려움 — 사람도 사전에 모르는 경우 많음

**개인 한계**

- 제 사용 방식(기능 구현 위주)의 한계일 수 있음
- 성능 크리티컬 영역은 여전히 사람이 직접 튜닝 중
